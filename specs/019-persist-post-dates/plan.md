# Implementation Plan: Persist Post Publish and Expiration Dates

**Branch**: `019-persist-post-dates` | **Date**: 2026-05-22 | **Spec**: [spec.md](spec.md)

---

## Summary

The `posts` table already has `publish_date` and `expire_date` columns, and the
frontend already renders date input fields. The feature wires these through
every layer: Domain entity → Application commands/results → raw-SQL repository
→ API contracts → React forms. It also applies the date window as a public
visibility filter across listing, detail, and search endpoints.

## 1. Technical Approach

This is a pure wiring task. The data model (DB columns), UI fields, and
repository pattern already exist. The implementation follows a strict
backend-first sequence:

1. Extend the domain entity with the two date properties and update the
   computed visibility property.
2. Extend Application commands, handlers (with date validation), and result
   DTOs.
3. Update the raw-SQL repository: INSERT/UPDATE parameters and
   SELECT/RETURNING projections.
4. Update API request/response DTOs and controller mappings.
5. Wire the existing frontend form fields to the API call.

At no point are new packages, patterns, or background jobs introduced.

## 3. Domain/Application Model Strategy

### Domain: `BlogPost`

- Add `DateTimeOffset? PublishDate { get; }` property.
- Add `DateTimeOffset? ExpirationDate { get; }` property.
- Update the private constructor to accept and store both values.
- Update `Create()` to accept optional `publishDate` and `expirationDate`.
- Update `Rehydrate()` to accept optional `publishDate` and `expirationDate`.
- Update `Update()` to accept optional `publishDate` and `expirationDate`, and
  carry the caller-supplied values (replacing whatever the post previously had).
- Update `IsPubliclyReadable`:
  ```
  IsPublic && IsAvailable
    && (PublishDate is null || PublishDate <= DateTimeOffset.UtcNow)
    && (ExpirationDate is null || ExpirationDate > DateTimeOffset.UtcNow)
  ```

No validation of date consistency lives in the domain; that is an
Application-layer concern.

### Application: commands

- `CreateBlogPostCommand`: add `DateTimeOffset? PublishDate` and
  `DateTimeOffset? ExpirationDate` record properties.
- `EditBlogPostCommand`: same additions.

### Application: handlers

- `CreateBlogPostHandler`:
  - Add date consistency validation before calling `BlogPost.Create()`.
  - Pass `publishDate` and `expirationDate` to `BlogPost.Create()`.
  - Map dates to `CreateBlogPostResult`.
- `EditBlogPostHandler`:
  - Add date consistency validation before calling `post.Update()`.
  - Pass `publishDate` and `expirationDate` to `post.Update()`.
  - Map dates to `EditBlogPostResult`.

### Application: results

All result and list item types gain nullable `PublishDate?` and `ExpirationDate?`
properties. Static `From()` factory methods are updated to map from the domain
entity's new properties.

## 4. Date Validation Strategy

**Location**: Application layer handlers only.

**Rule**: When both `PublishDate` and `ExpirationDate` are provided,
`ExpirationDate > PublishDate`.

**Error contract**: Return a validation failure that the API maps to
`HTTP 400` with a ProblemDetails body. The error key is `"expirationDate"` and
the message is `"Expiration date must be after publish date."`.

This matches the existing validation failure pattern used for title/content
validation in `CreateBlogPostHandler` and `EditBlogPostHandler`. No new error
type is introduced.

---

## 5. Visibility Rule Strategy

### Rule (full)
A post is publicly visible when:
1. `IsPublic = true`
2. `IsAvailable = true`
3. `PublishDate IS NULL OR PublishDate <= NOW()`
4. `ExpirationDate IS NULL OR ExpirationDate > NOW()`

### Domain property (authoritative in-memory check)
`BlogPost.IsPubliclyReadable` evaluates all four conditions. Application
handlers that already call `IsPubliclyReadable` gain date awareness for free.

### Repository SQL (performance mirror)
`ListPublicReadAsync` SQL gains a WHERE clause:
```sql
WHERE p.public_post = TRUE
  AND p.available = TRUE
  AND (p.publish_date IS NULL OR p.publish_date <= NOW())
  AND (p.expire_date IS NULL OR p.expire_date > NOW())
```

`SearchPublicReadAsync` SQL extends its existing WHERE clause:
```sql
WHERE p.available = TRUE
  AND (
    (   p.public_post = TRUE
        AND (p.publish_date IS NULL OR p.publish_date <= NOW())
        AND (p.expire_date IS NULL OR p.expire_date > NOW())
    )
    OR (@requesting_user_id IS NOT NULL AND p.user_id = @requesting_user_id)
  )
  AND (text search conditions)
```

`GetPublicReadByIdAsync` SQL gains the same date conditions in its WHERE
clause.

The Application handler's `IsPubliclyReadable` check remains as the
authoritative gate even when the SQL has filtered; this prevents incorrect
results if the clock crosses a boundary between query and in-memory evaluation.

### Authenticated search ownership bypass
Authenticated users see their own posts regardless of the visibility window.
This is already implemented in `SearchPublicReadAsync` via the
`OR p.user_id = @requesting_user_id` condition; the new date conditions are
structured so the ownership bypass sits outside the date window check (see SQL
above).

## 6. Database/Schema Impact

**No DDL change required.** Columns `publish_date` and `expire_date` already
exist in `005-create-posts.sql`.

**Seed data update** (if seed scripts exist): add representative rows as
described in `data-model.md` to support integration test scenarios without
in-test fixture setup.

**Existing INSERT behavior change**: The current expression
`CASE WHEN @public_post THEN NOW() ELSE NULL END` for `publish_date` must be
replaced with a `@publish_date` parameter. Existing rows in a development
database may have `publish_date = creation_date` (set to NOW() at creation);
these rows remain visible because `publish_date <= NOW()`.

## 7. Repository/Raw SQL Strategy

### INSERT statement update

Replace:
```sql
INSERT INTO posts (
    user_id, post_category_id, title, description, content,
    available, public_post, publish_date, creation_user_id, update_user_id)
VALUES (
    @user_id, @post_category_id, @title, @description, @content,
    @available, @public_post,
    CASE WHEN @public_post THEN NOW() ELSE NULL END,
    @creation_user_id, @update_user_id)
RETURNING id, user_id, post_category_id, title, description, content,
    public_post, available;
```

With:
```sql
INSERT INTO posts (
    user_id, post_category_id, title, description, content,
    available, public_post, publish_date, expire_date,
    creation_user_id, update_user_id)
VALUES (
    @user_id, @post_category_id, @title, @description, @content,
    @available, @public_post, @publish_date, @expire_date,
    @creation_user_id, @update_user_id)
RETURNING id, user_id, post_category_id, title, description, content,
    public_post, available, publish_date, expire_date;
```

### UPDATE statement update

Add to SET clause:
```sql
publish_date = @publish_date,
expire_date = @expire_date,
```
Add to RETURNING:
```sql
RETURNING ..., publish_date, expire_date;
```

### SELECT statement updates

All SELECT statements that project post columns must add:
```sql
p.publish_date, p.expire_date
```
after the existing 8 columns. The tags subquery shifts to ordinal 10.

### `MapPost` / `MapPostFromReader` updates

Use named ordinal lookups via `reader.GetOrdinal("column_name")` to avoid
positional fragility:
- `publish_date` → `DateTimeOffset?` (null if `reader.IsDBNull`)
- `expire_date` → `DateTimeOffset?` (null if `reader.IsDBNull`)

Pass to `BlogPost.Rehydrate(publishDate:..., expirationDate:...)`.

### `BuildPostInsertCommand` / `BuildPostUpdateCommand` helpers

Add `@publish_date` and `@expire_date` `NpgsqlParameter<DateTimeOffset?>` to
the command builder methods (using `NpgsqlDbType.TimestampTz`).

## 9. Frontend Integration Strategy

### Type changes (`post.types.ts`)

1. Add `publishDate?` and `expirationDate?` to `PostMutationRequest`:
   ```typescript
   publishDate?: string | null
   expirationDate?: string | null
   ```

2. Update `PostEditorDraft` to rename `expireDate` → `expirationDate` for
   consistency with the API contract. Update all references.

### API helper (`owned-posts.api.ts`)

The `createPost()` and `updatePost()` functions already accept
`PostMutationRequest`. No signature change needed after the type update above.

When submitting, convert empty string date fields to `null` (so the API
receives `null` rather than an empty string):
```typescript

expirationDate: draft.expirationDate || null,
```

### `CreatePostPage.tsx`

- Initial draft already includes `publishDate: ''` and `expireDate: ''`.
- Rename `expireDate` → `expirationDate` in initial state and draft type.
- Include both fields in the `createPost(...)` call:
  ```typescript
  createPost({
    ...draft,
    tags: parseTags(draft.tags),
    publishDate: draft.publishDate || null,
    expirationDate: draft.expirationDate || null,
  }, user.token)
  ```

### `EditPostPage.tsx`

- When loading the existing post, populate form state with `post.publishDate`
  and `post.expirationDate` from the API response (format for datetime-local
  input as needed, e.g. truncate the `Z` suffix).
- Include both fields in the `updatePost(...)` call as above.

### `PostForm.tsx`

- The `publishDate` and `expireDate` inputs already render. Rename the
  `expireDate` reference to `expirationDate` in the form handler and form
  fields.
- Remove or update the "Metadata fields are ready in the UI and awaiting
  backend support" UI status note (lines 165-173) now that backend is wired.
- Add an inline error display for the `expirationDate` API validation error
  (400 response).

### Date format note

`<input type="datetime-local">` produces `YYYY-MM-DDTHH:mm` (no `Z`). The API
expects ISO 8601 UTC. Two options:
- Append `":00Z"` to the submitted string if not already ending in `Z`.
- Parse with `new Date(value).toISOString()`.

The recommended approach: convert to ISO UTC string before the API call in the
page components.

## 14. Risks and Trade-offs

| Risk | Impact | Mitigation |
|------|--------|-----------|
| `IsPubliclyReadable` uses `DateTimeOffset.UtcNow` — unit tests are time-sensitive | Low — tests use dates far in the future/past | Document test convention: scheduled = today + 30 days; expired = today - 1 day |
| SQL `WHERE NOW()` is evaluated at query time; Application `IsPubliclyReadable` is evaluated slightly later | Negligible — sub-millisecond window for a single request | Accepted; no clock-skew mitigation needed at interview scale |
| Positional column ordinals in `MapPost` shift after adding two columns | High if missed | Use named ordinal lookups (`reader.GetOrdinal`) in mapper functions |
| Existing rows with `publish_date = creation_date` (auto-set by old INSERT logic) | None — those dates are in the past, so posts remain visible | No data migration needed |
| Frontend `<input type="datetime-local">` submits without `Z` suffix | Medium — server may reject or mis-parse | Normalize to ISO UTC string in page before API call |
| `expireDate` → `expirationDate` rename in frontend draft type | Low — localized to 3 files | Update all references in one pass; TypeScript compile errors guide completeness |
